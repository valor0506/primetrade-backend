# PrimeTrade API — Backend Assignment

A secure, production-structured REST API for trade management, built with **FastAPI**, **PostgreSQL**, and **JWT authentication**. Includes a fully functional vanilla JS frontend dashboard.

---

## Tech Stack

| Layer       | Technology                          |
|-------------|-------------------------------------|
| Backend     | FastAPI (Python 3.12)               |
| Database    | PostgreSQL 16 + SQLAlchemy 2.0 ORM  |
| Auth        | JWT (python-jose) + bcrypt hashing  |
| Validation  | Pydantic v2                         |
| Frontend    | Vanilla JS / HTML / CSS (no build step) |
| Deployment  | Docker + Docker Compose             |
| API Docs    | Swagger UI (auto-generated by FastAPI) |

## Database Schema

### `users`
| Column            | Type        | Notes                      |
|-------------------|-------------|----------------------------|
| id                | integer PK  | Auto-increment             |
| email             | varchar     | Unique, indexed            |
| username          | varchar     | Unique, indexed            |
| hashed_password   | varchar     | bcrypt, never stored plain |
| role              | enum        | `user` or `admin`          |
| is_active         | boolean     | Soft deactivation          |
| created_at        | timestamptz | Server default             |

### `trades`
| Column       | Type        | Notes                           |
|--------------|-------------|---------------------------------|
| id           | integer PK  | Auto-increment                  |
| owner_id     | integer FK  | References `users.id`           |
| symbol       | varchar     | e.g. `BTC/USDT`, uppercased     |
| trade_type   | enum        | `buy` or `sell`                 |
| quantity     | float       | Must be > 0                     |
| price        | float       | Entry price, must be > 0        |
| status       | enum        | `open`, `closed`, `cancelled`   |
| notes        | varchar     | Optional, max 500 chars         |
| created_at   | timestamptz | Server default                  |
| updated_at   | timestamptz | Updated on PATCH                |

## Setup — Local (without Docker)

### 1. Clone and install dependencies

```bash
git clone https://github.com/YOUR_USERNAME/primetrade-backend.git
cd primetrade-backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Set up PostgreSQL

Create a database named `primetrade_db` and a user with access, then:

```bash
cp .env.example .env
# Edit .env with your DATABASE_URL and a strong SECRET_KEY
```

### 3. Run the server

```bash
uvicorn main:app --reload
```

Tables are auto-created on first startup via `Base.metadata.create_all()`.

## API Reference

Interactive docs available at: **http://localhost:8000/docs**

### Authentication — `/api/v1/auth`

| Method | Endpoint         | Auth     | Description              |
|--------|------------------|----------|--------------------------|
| POST   | `/register`      | None     | Register new user        |
| POST   | `/login`         | None     | Login, returns JWT       |
| GET    | `/me`            | Bearer   | Get current user profile |

**Register request:**
```json
{
  "email": "trader@example.com",
  "username": "trader1",
  "password": "Secure123"
}
```

**Login response:**
```json
{
  "access_token": "eyJ...",
  "token_type": "bearer",
  "user": { "id": 1, "email": "...", "username": "...", "role": "user" }
}
```

---

### Trades — `/api/v1/trades`

All endpoints require `Authorization: Bearer <token>`

| Method | Endpoint         | Role       | Description                  |
|--------|------------------|------------|------------------------------|
| POST   | `/`              | user/admin | Create a new trade           |
| GET    | `/`              | user/admin | List own trades (filterable) |
| GET    | `/all`           | admin only | List all trades platform-wide|
| GET    | `/{id}`          | user/admin | Get single trade             |
| PATCH  | `/{id}`          | user/admin | Update trade fields          |
| DELETE | `/{id}`          | user/admin | Delete a trade               |

**Create trade request:**
```json
{
  "symbol": "BTC/USDT",
  "trade_type": "buy",
  "quantity": 0.5,
  "price": 65000,
  "notes": "Breakout from weekly resistance"
}
```

**Filter trades by status:**
```
GET /api/v1/trades/?status=open
```

## Security Practices

- **Passwords** — bcrypt hashed with `passlib`, never stored or logged in plain text
- **JWT** — HS256 signed tokens, configurable expiry via env variable
- **Input validation** — Pydantic v2 validates all inputs; symbol sanitized to uppercase, passwords require uppercase + digit
- **Role enforcement** — `require_admin` dependency injected at the route level; users can only access their own trades
- **CORS** — configurable origins via middleware (tighten in production)
- **Global error handler** — catches unhandled exceptions, never leaks stack traces

## Frontend

Open `frontend/index.html` directly in a browser (no build step). Ensure the API is running at `http://localhost:8000`.

**Features:**
- Register / Login with JWT token stored in `localStorage`
- Dashboard with trade stats (total, open, buys, sells)
- Full trade CRUD — create, filter, edit, delete
- Role-aware UI — admin sees "All Trades" and "User Management" panels
- Toast notifications and inline error messages from API responses

## Scalability Notes

### Current architecture
Single service, single database — suitable for early-stage deployments and development.

### Horizontal scaling path
1. **Stateless API** — JWT-based auth means any API instance can handle any request. Drop a load balancer (NGINX / AWS ALB) in front and scale replicas freely.
2. **Connection pooling** — Replace direct psycopg2 connections with PgBouncer or SQLAlchemy's `pool_size` / `max_overflow` for high-concurrency scenarios.
3. **Caching** — Add Redis (via `fastapi-cache2`) for frequently-read endpoints like `/trades/all` to reduce DB load.
4. **Async DB** — Migrate from sync SQLAlchemy to `asyncpg` + `sqlalchemy[asyncio]` for non-blocking I/O under heavy load.
5. **Microservices** — Auth, trade management, and user management are already modular routers. Extracting them into separate services (with shared JWT secret or a dedicated auth service) is a natural next step.
6. **Background tasks** — Long-running operations (e.g., trade analytics, reports) can be offloaded to Celery + Redis without touching the API layer.
7. **Containerization** — Docker Compose for local; Kubernetes (EKS/GKE) for production with auto-scaling and health probes.
